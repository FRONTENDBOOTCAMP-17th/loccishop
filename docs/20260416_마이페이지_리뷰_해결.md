# 마이페이지 리뷰 해결 가이드

> 작성 날짜: 2026-04-16
> 원본 리뷰: [20260416_마이페이지_리뷰.md](./20260416_마이페이지_리뷰.md)

---

## 1. 먼저, 어떤 순서로 고치면 좋을까요?

초보 개발자라면 모든 문제를 한 번에 잡으려 하지 말고, 아래 순서로 처리하는 것이 좋습니다.

1. 마이페이지가 배포에서 열리도록 경로와 빌드 설정부터 맞추기
2. 샘플 데이터와 실제 데이터 영역을 분리하기
3. 사이드바 활성 상태를 한 곳에서만 관리하도록 정리하기
4. 모바일 반응형을 TailwindCSS로 보강하기
5. 중복된 렌더 함수를 하나의 공통 함수로 합치기

이 순서를 추천하는 이유는 간단합니다.

- 1번은 화면 자체가 안 열릴 수 있는 문제입니다
- 2번은 사용자에게 잘못된 정보를 보여줄 수 있는 문제입니다
- 3번은 클릭할수록 UI가 어긋날 수 있는 문제입니다
- 4번은 실제 사용성 문제입니다
- 5번은 유지보수 문제입니다

---

## 2. 마이페이지가 배포에서도 열리게 고치기

### 문제 다시 보기

현재 코드는 아래 두 가지에 강하게 묶여 있습니다.

1. `vite.config.js`에 마이페이지 엔트리가 없다
2. 각 섹션을 `/src/pages/...html` 경로로 직접 `fetch`한다

이 구조는 개발 중에는 보일 수 있지만, 빌드와 배포에서는 매우 약합니다.

### Step 1. `vite.config.js`에 `mypage` 엔트리를 추가하세요

먼저 마이페이지 HTML이 실제 빌드 대상이 되어야 합니다.

예시:

```js
build: {
  rollupOptions: {
    input: {
      main: resolve(__dirname, "index.html"),
      login: resolve(__dirname, "/src/pages/login/index.html"),
      mypage: resolve(__dirname, "/src/pages/mypage/index.html"),
    },
  },
}
```

여기서 중요한 포인트는:

- 페이지를 새로 만들었으면
- Vite가 그 페이지를 빌드 대상으로 알고 있어야 한다

입니다.

### Step 2. `/src/pages/...html`을 `fetch`하는 구조를 멈추세요

현재 방식:

```js
const response = await fetch("/src/pages/mypage/components/my-info.html");
```

이 구조는 "현재 폴더 위치"에 너무 의존합니다.

초보 개발자에게는 아래 방식이 더 안전합니다.

- 방법 A. 각 섹션을 JavaScript 모듈로 만들기
- 방법 B. `index.html` 안에 `<template>`으로 두고 복사해서 쓰기

이 프로젝트에서는 **방법 A**를 추천합니다. 이유는 파일 역할이 더 분명해지기 때문입니다.

### Step 3. 섹션별 템플릿 함수를 만드세요

예를 들어:

```js
// src/pages/mypage/components/myInfoTemplate.js
export function getInfoTemplate() {
  return `
    <div class="bg-white-solid rounded-sm border border-cararra p-6 md:p-10">
      <h3 id="info-summary-name" class="text-2xl md:text-3xl font-light">-</h3>
      <p id="info-summary-email" class="text-sm text-empress/70">-</p>
    </div>
  `;
}
```

그리고 렌더 함수는 이렇게 단순해집니다.

```js
import { getInfoTemplate } from "../components/myInfoTemplate.js";

export function renderInfo() {
  const container = document.querySelector("#mypage-content");
  if (!container) return;

  container.innerHTML = getInfoTemplate();
}
```

이 방식의 장점:

- 파일 경로를 네트워크로 다시 읽지 않습니다
- 번들링에 더 자연스럽게 포함됩니다
- "레이아웃"과 "데이터 채우기"를 분리하기 쉽습니다

### Step 4. 빌드 검증은 꼭 하세요

수정 후에는 아래를 꼭 확인해야 합니다.

```bash
npm run build
```

다만 이번 저장소는 현재 `vite.config.js`의 다른 엔트리 경로 문제로 빌드가 먼저 실패하고 있습니다.

- 현재 실패 원인: `src/pages/signup/index.html` 경로 불일치

즉, 마이페이지만 보지 말고 **전체 엔트리 경로도 같이 정리**해야 합니다.

## 3. 샘플 데이터와 실제 데이터를 분리하기

### 문제 다시 보기

현재 화면에는 샘플 값이 실제 값처럼 들어 있습니다.

```html
홍길동님
hong@naver.com
15,000P
4.7
3 items
```

이 값들은 디자인 확인용으로는 편하지만, 실제 서비스 화면에 그대로 보이면 안 됩니다.

### Step 1. 기본 마크업에는 "비어 있는 상태"만 두세요

좋은 기본값 예시:

- `-`
- `0건`
- `불러오는 중...`
- `등록된 정보가 없습니다`

예시:

```html
<h3 id="info-summary-name">-</h3>
<p id="info-summary-email">-</p>
<p id="info-summary-point">0 <span>P</span></p>
```

### Step 2. 실제 값은 JavaScript에서 `textContent`로 넣으세요

예시:

```js
function setText(selector, value) {
  const el = document.querySelector(selector);
  if (!el) return;
  el.textContent = value;
}

function bindUserInfo(user) {
  setText("#info-summary-name", user.name ? `${user.name}님` : "이름 정보 없음");
  setText("#info-summary-email", user.email ?? "이메일 정보 없음");
  setText("#info-summary-join-date", user.joinedAt ?? "가입일 정보 없음");
}
```

왜 `textContent`를 추천하나요?

- 문자열을 안전하게 넣을 수 있습니다
- HTML로 해석되지 않습니다
- 나중에 데이터 출처가 API로 바뀌어도 더 안전합니다

### Step 3. 레이아웃 파일에는 "디자인"만 남기세요

초보 개발자가 가장 많이 헷갈리는 부분이 이것입니다.

- 디자인 확인용 샘플 텍스트
- 실제 사용자 데이터

이 둘을 한 파일에 같이 넣어두면 나중에 반드시 헷갈립니다.

그래서 역할을 이렇게 나누는 것이 좋습니다.

- 템플릿: 박스, 제목, 영역, `id`
- 데이터 바인딩 함수: 실제 값 채우기

이렇게 나누면 응집도는 높아지고, 샘플 문자열에 대한 결합도는 낮아집니다.

## 4. 사이드바 활성 상태를 한 곳에서만 관리하기

### 문제 다시 보기

현재는:

- HTML이 초기 활성 스타일을 일부 가지고 있고
- JavaScript가 클릭 후 일부 클래스만 토글합니다

이 구조에서는 상태가 바뀔수록 클래스 조합이 꼬이기 쉽습니다.

### Step 1. 공통 클래스와 상태별 클래스를 분리하세요

예시:

```js
const BASE_MENU_CLASS =
  "group w-full flex items-center justify-between py-3 text-sm border-b transition-colors";

const ACTIVE_MENU_CLASS = "text-ferra font-semibold border-cararra bg-merino";

const INACTIVE_MENU_CLASS =
  "text-empress border-cararra/30 hover:text-dark-woody";
```

### Step 2. 버튼 전체 클래스를 다시 써주세요

부분 토글보다 초보자에게 더 쉬운 방법은 "상태에 맞는 전체 클래스"를 다시 넣는 것입니다.

예시:

```js
function updateSidebarUI(activeMenu) {
  const buttons = document.querySelectorAll("#mypage-nav button[data-menu]");

  buttons.forEach((button) => {
    const isActive = button.dataset.menu === activeMenu;
    button.className = `${BASE_MENU_CLASS} ${isActive ? ACTIVE_MENU_CLASS : INACTIVE_MENU_CLASS}`;
  });
}
```

이 방식의 장점:

- 활성 상태 기준이 한 곳에 있습니다
- 이전 상태 클래스가 남는 문제를 줄일 수 있습니다
- HTML과 JavaScript가 서로 다른 스타일 규칙을 갖지 않게 됩니다

### Step 3. 선택 상태를 코드로 명확히 표현하세요

가능하면 `aria-current` 같은 속성도 같이 넣어주세요.

예시:

```js
button.setAttribute("aria-current", isActive ? "page" : "false");
```

이건 접근성에도 도움이 되고, 나중에 상태를 디버깅할 때도 편합니다.

## 5. 모바일 반응형은 TailwindCSS로 해결하세요

### 문제 다시 보기

현재는 고정 열 구조가 많아서 모바일에서 폭이 줄어들면 바로 답답해질 수 있습니다.

이 문제는 CSS 파일을 새로 만들지 않고도 해결할 수 있습니다.

### Step 1. 바깥 레이아웃을 모바일 우선으로 바꾸세요

예시:

```html
<main class="max-w-[1280px] mx-auto w-full px-4 py-10 sm:px-6 lg:px-10 lg:py-20 flex flex-col lg:flex-row gap-8 lg:gap-20">
  <aside class="w-full lg:w-56 lg:flex-shrink-0"></aside>
  <section id="mypage-content" class="flex-1 min-h-[600px]"></section>
</main>
```

핵심은:

- 기본은 세로 배치
- 큰 화면에서만 가로 배치

입니다.

### Step 2. 정보 카드와 위시리스트 그리드에 반응형 열 수를 주세요

예시:

```html
<div class="grid grid-cols-1 md:grid-cols-2 gap-8 md:gap-x-16 md:gap-y-12">
```

```html
<div id="wish-list-container" class="grid grid-cols-1 sm:grid-cols-2 xl:grid-cols-3 gap-x-12 gap-y-20"></div>
```

이렇게 하면 작은 화면에서는 1열, 화면이 넓어지면 2열, 3열로 자연스럽게 바뀝니다.

### Step 3. 주문 목록처럼 가로 칸이 많은 영역은 `overflow-x-auto`도 고려하세요

예시:

```html
<div class="overflow-x-auto">
  <div class="min-w-[640px] grid grid-cols-[1fr_2fr_1fr_1fr]">
    ...
  </div>
</div>
```

이 방식은 억지로 글자를 줄여서 망가뜨리는 것보다 훨씬 안전합니다.

### Step 4. 별도 CSS보다 Tailwind 반응형 유틸리티를 먼저 떠올리세요

초보 단계에서는 아래 순서로 생각하는 습관이 좋습니다.

1. Tailwind 클래스만으로 해결 가능한가?
2. 반응형 접두사(`sm:`, `md:`, `lg:`)로 해결 가능한가?
3. 정말 안 되면 그때 최소한의 CSS를 쓸 것인가?

이번 경우는 대부분 1번과 2번으로 해결 가능합니다.

## 6. 렌더 함수 4개를 하나의 공통 함수로 합치기

### 문제 다시 보기

현재 `renderInfo`, `renderOrderList`, `renderWishList`, `renderReviewList`는 거의 같은 코드입니다.

이런 코드는 초반에는 괜찮아 보여도, 나중에 수정 비용이 빠르게 커집니다.

### Step 1. 메뉴와 렌더 함수를 객체로 매핑하세요

예시:

```js
import { getInfoTemplate } from "../components/myInfoTemplate.js";
import { getOrderTemplate } from "../components/myOrderTemplate.js";
import { getWishTemplate } from "../components/myWishTemplate.js";
import { getReviewTemplate } from "../components/myReviewTemplate.js";

const SECTION_TEMPLATES = {
  info: getInfoTemplate,
  order: getOrderTemplate,
  wish: getWishTemplate,
  review: getReviewTemplate,
};
```

### Step 2. 공통 렌더 함수를 만드세요

예시:

```js
function renderSection(menu) {
  const container = document.querySelector("#mypage-content");
  const template = SECTION_TEMPLATES[menu];

  if (!container || !template) return;

  try {
    container.innerHTML = template();
  } catch (error) {
    container.innerHTML = `
      <p class="p-10 text-center text-error-red">
        화면을 불러오는 중 오류가 발생했습니다.
      </p>
    `;
    console.error(error);
  }
}
```

### Step 3. 클릭 이벤트도 더 단순하게 바꾸세요

예시:

```js
nav.addEventListener("click", (event) => {
  const button = event.target.closest("button[data-menu]");
  if (!button) return;

  const menu = button.dataset.menu;
  updateSidebarUI(menu);
  renderSection(menu);
});
```

이렇게 바꾸면 좋은 점:

- 같은 에러 UI를 재사용할 수 있습니다
- 메뉴가 늘어나도 `switch`가 길어지지 않습니다
- 수정할 곳이 줄어듭니다

즉, **응집도는 높아지고 결합도는 낮아집니다.**

## 7. 마지막 체크리스트

수정이 끝나면 아래를 꼭 확인해보세요.

1. 로그인 후 마이페이지 링크가 실제로 열리는가
2. 페이지 최초 진입 시 가짜 이름과 가짜 수치가 보이지 않는가
3. 메뉴를 여러 번 눌러도 활성 스타일이 일관적인가
4. 모바일 폭에서도 옆으로 과하게 밀리지 않는가
5. `npm run build`가 성공하는가

이 다섯 가지가 통과되면, 이번 마이페이지 작업은 "디자인 초안" 단계에서 "운영 가능한 기본 구조" 단계로 올라간 것입니다.

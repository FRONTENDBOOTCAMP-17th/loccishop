## 0. 이번 리뷰 범위

이번 리뷰는 새로 `pull`된 변경 중 아래 파일을 중심으로 봤습니다.

- `admin/js/products.js`
- `index.html`
- `src/components/ui/carouselSlide.js`
- `src/js/api/carousel/index.js`
- `src/js/main.js`
- `vite.config.js`

리뷰 기준은 아래와 같습니다.

- 초보 개발자도 읽고 수정할 수 있는가
- 한 파일이 한 가지 역할에 집중하고 있는가
- 다른 파일이나 데이터 구조에 너무 강하게 묶여 있지 않은가
- CSS는 꼭 필요한 곳만 쓰고, 가능하면 TailwindCSS로 표현하고 있는가

## 1. 먼저, 잘한 점

### 배너를 모바일/데스크톱으로 나눠 가져오려는 시도

`src/js/main.js`에서 `mobile`, `sub1` 두 위치의 배너를 나눠서 가져오고, 화면 크기에 따라 다른 세트를 보여주려는 방향은 좋았습니다.

이 접근의 장점은:

- 화면 크기별로 다른 이미지를 줄 수 있습니다
- 한 장의 이미지를 억지로 늘리거나 줄이는 것보다 관리가 쉽습니다
- 배너 데이터가 늘어나도 API 쪽에서 분리해서 관리할 수 있습니다

즉, 아이디어 자체는 충분히 좋은 방향입니다.

### 슬라이드 폭을 HTML과 컴포넌트에서 맞추려는 점

`index.html`과 `carouselSlide.js`에서 같은 폭 기준을 사용하려고 맞춘 점도 좋았습니다.

- `w-[min(450px,90vw)]`
- `md:w-[450px]`
- `lg:w-[914px]`
- `xl:w-[942px]`

이런 식으로 기준을 통일하면 "여기는 914인데 저기는 900" 같은 어긋남이 줄어듭니다.

### 관리자 상품 목록에 카테고리 필터를 붙인 시도

관리자 페이지에서 검색어만 있던 상태보다 카테고리 필터가 추가된 것은 실제 사용성을 올리는 변화입니다.

특히 아래처럼 `loadProducts()`의 인자를 확장한 것은 함수 이름과 역할이 자연스럽습니다.

```js
loadProducts(page, keyword, categoryId)
```

기능을 무작정 늘리는 것이 아니라, 기존 흐름 안에서 확장했다는 점은 좋았습니다.

## 2. 개선이 필요한 점

### 2-1. 배너 데이터가 비어 있으면 메인 페이지 캐러셀이 깨질 수 있어요 (중요도: 높음)

**파일**

- `src/js/main.js:160-161`
- `src/js/main.js:201`
- `src/js/main.js:259-275`
- `src/js/main.js:325-330`

현재 코드는 배너 API 호출이 실패하면 `[]`를 넣도록 되어 있습니다.

```js
fetchCarousels("mobile").catch(() => [])
fetchCarousels("sub1").catch(() => [])
```

그런데 아래 코드에서는 `slides`가 비어 있어도 그대로 첫 번째/마지막 슬라이드를 만들려고 합니다.

```js
track.append(createCarouselSlide(slides[slides.length - 1]));
slides.forEach((s) => track.append(createCarouselSlide(s)));
track.append(createCarouselSlide(slides[0]));
```

`slides.length`가 `0`이면:

- `slides[slides.length - 1]`는 `undefined`
- `slides[0]`도 `undefined`
- `createCarouselSlide(undefined)`가 호출됩니다

그리고 `updateNav()`에서는 `slides.length`가 `0`인데도 나머지 연산을 하고 있습니다.

```js
slides[(realIdx - 1 + total) % total]
```

여기서 `total`이 `0`이면 계산 자체가 깨집니다.

#### 왜 문제인가요?

- 배너 API가 잠깐만 실패해도 메인 페이지 핵심 영역이 깨질 수 있습니다
- "데이터가 없을 수도 있다"는 상황을 코드가 감당하지 못하고 있습니다
- 기존에는 임시 슬라이드 데이터가 있었는데, 이번 변경에서 그 안전장치가 사라졌습니다

초보 개발자에게 아주 중요한 습관이 하나 있습니다.

> "API는 항상 성공하지 않는다."

성공하는 경우만 생각하면 실제 서비스에서 자주 깨집니다.

#### 어떻게 생각하면 쉬울까요?

배열이 비어 있는 상태는 "예외 상황"이 아니라 "자주 올 수 있는 정상 상황"입니다.

- 서버 점검 중일 수도 있고
- 아직 배너를 하나도 등록하지 않았을 수도 있고
- 모바일 배너만 비어 있을 수도 있습니다

즉, 캐러셀 코드는 "슬라이드가 0개, 1개, 여러 개"인 경우를 모두 안전하게 처리해야 합니다.

### 2-2. 배너 API 데이터를 `innerHTML`에 바로 넣고 있어요 (중요도: 높음)

**파일**

- `src/js/api/carousel/index.js:7-13`
- `src/components/ui/carouselSlide.js:6-15`

현재 배너 데이터는 API에서 받아옵니다.

```js
return list.map(({ imageUrl, title, linkUrl }) => ({
  imageUrl,
  name: title,
  href: linkUrl ?? "#",
}));
```

그리고 화면에서는 `innerHTML`로 그대로 삽입하고 있습니다.

```js
el.innerHTML = `
  <figure ...>
    <img
      src="${imageUrl}"
      alt="${name}"
      ...
    />
  </figure>
`;
```

#### 왜 문제인가요?

`innerHTML`은 문자열을 "HTML 코드"로 해석합니다.

그래서 값 안에 예상하지 못한 문자가 들어오면:

- 마크업이 깨질 수 있고
- 악의적인 코드가 섞일 수 있고
- 나중에 버튼이나 링크가 추가되면 더 위험해질 수 있습니다

특히 이번 코드는 정적인 상수가 아니라 **관리 가능한 배너 데이터**를 API에서 받아옵니다.  
즉, 이제는 "내가 직접 쓴 문자열"이 아니라 "외부 데이터"입니다.

초보자 입장에서는 이렇게 기억하면 됩니다.

> 내가 만든 고정 문자열은 비교적 안전합니다.  
> 서버에서 온 문자열은 항상 조심해야 합니다.

#### 왜 이게 구조 문제이기도 한가요?

`fetchCarousels()`는 데이터를 가져오는 역할이고, `createCarouselSlide()`는 화면을 만드는 역할입니다.  
그런데 화면을 만드는 함수가 문자열 HTML에 너무 의존하면 나중에:

- 텍스트만 안전하게 넣고 싶은 경우
- 링크 주소를 검증하고 싶은 경우
- 접근성 속성을 추가하고 싶은 경우

수정이 어려워집니다.

즉, DOM을 직접 만드는 방식이 **응집도는 높이고**, 데이터 문자열에 대한 **결합도는 낮춰줍니다.**

### 2-3. 관리자 상품 카테고리 필터가 하드코딩되어 있어서 결합도가 높아요 (중요도: 중간)

**파일**

- `admin/js/products.js:33-48`
- `admin/js/products.js:94-96`
- `admin/js/categories.js:3-14`
- `admin/js/categories.js:70-85`

이번에 추가된 카테고리 필터는 아래처럼 값이 코드에 직접 적혀 있습니다.

```html
<option value="2">핸드 크림</option>
<option value="3">핸드 워시 & 솝</option>
<option value="4">핸드 & 네일 케어</option>
...
<option value="23">고체 솝</option>
```

문제는 이 프로젝트 안에 이미 카테고리 API가 있다는 점입니다.

```js
export async function loadCategories() {
  return fetchAPI("/categories");
}
```

즉, 카테고리 정보의 진짜 원본은 서버인데, 상품 관리 화면에서 다시 숫자와 이름을 직접 적어버렸습니다.

#### 왜 문제인가요?

- 카테고리 이름이 바뀌면 `products.js`도 수정해야 합니다
- 카테고리 ID가 달라지면 필터가 조용히 잘못 동작할 수 있습니다
- 운영자가 새 카테고리를 만들어도 드롭다운에는 안 보입니다
- 상품 등록 모달은 여전히 `카테고리 ID`를 숫자로 직접 입력해야 해서 실수하기 쉽습니다

이건 초보자에게 아주 좋은 구조 연습 포인트입니다.

> 데이터의 원본은 한 곳이어야 합니다.

카테고리의 원본이 서버라면:

- 목록 필터도 그 데이터를 써야 하고
- 등록 폼도 그 데이터를 써야 하고
- 이름 표시도 그 데이터를 써야 합니다

그래야 한 군데가 바뀌어도 전체가 같이 맞춰집니다.

#### 응집도/결합도로 보면?

- 지금은 `products.js`가 "상품 관리"만 아는 것이 아니라 "카테고리 ID 체계"까지 외우고 있습니다
- 즉, 한 파일이 너무 많은 것을 알고 있습니다
- 이런 상태가 바로 **불필요한 결합**입니다

상품 화면은 "카테고리를 불러와서 보여준다"까지만 알면 충분합니다.  
카테고리 구조 자체는 카테고리 모듈이 책임지는 편이 더 좋습니다.

### 2-4. `vite.config.js`의 proxy 설정이 현재 코드와 연결되지 않아요 (중요도: 낮음)

**파일**

- `vite.config.js:12-17`
- `src/js/api/client.js:3`
- `admin/js/api.js:1`

이번에 proxy 설정이 추가됐습니다.

```js
proxy: {
  "/api": {
    target: "https://api.fullstackfamily.com",
    changeOrigin: true,
  },
},
```

그런데 실제 API 호출 코드는 이렇게 절대 주소를 직접 사용하고 있습니다.

```js
const BASE_URL = "https://api.fullstackfamily.com/api/loccishop/v1";
```

Vite proxy는 보통 아래처럼 상대 경로로 요청할 때 의미가 있습니다.

```js
fetch("/api/loccishop/v1/products")
```

하지만 지금 구조에서는 이미 완성된 절대 URL로 요청하기 때문에 proxy를 타지 않습니다.

#### 왜 문제인가요?

- 설정은 있는데 실제로는 동작하지 않는 코드가 됩니다
- 초보 개발자는 "proxy를 넣었는데 왜 안 되지?" 하고 더 헷갈립니다
- 나중에 환경 변수를 붙이려 할 때 기준이 흔들립니다

이건 큰 버그는 아니지만, **죽어 있는 설정(dead config)** 은 초보자에게 특히 혼란을 줍니다.

## 3. 정리

이번 변경은 방향 자체는 좋았습니다.

- 배너를 API와 화면 크기에 맞춰 분리하려는 시도
- 관리자 상품 목록에 카테고리 필터를 넣은 시도
- 레이아웃 폭을 화면 크기별로 맞추려는 시도

하지만 실제 서비스 코드에서는 아래 2가지를 꼭 더 챙겨야 합니다.

1. **빈 데이터와 실패 상황을 먼저 처리하기**
2. **원본 데이터는 한 곳에서만 관리하기**

이 두 가지만 잘 지켜도 코드의 안정성과 유지보수성이 크게 좋아집니다.

자세한 해결 방법은 [20260415_리뷰_해결.md](./20260415_리뷰_해결.md)에 정리했습니다.

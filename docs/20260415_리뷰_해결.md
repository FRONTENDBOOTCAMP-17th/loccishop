## 1. 먼저 우선순위를 정해봅시다

초보 개발자라면 모든 문제를 한 번에 고치려고 하기보다, 아래 순서로 처리하는 것이 좋습니다.

1. 메인 페이지가 깨지는 문제부터 막기
2. `innerHTML`로 외부 데이터를 넣는 부분을 안전하게 바꾸기
3. 카테고리 데이터를 한 곳에서 관리하도록 구조 정리하기
4. 개발 설정(proxy)을 실제 코드와 맞추기

이 순서를 추천하는 이유는 간단합니다.

- 1번은 서비스가 실제로 깨질 수 있는 문제입니다
- 2번은 보안 문제입니다
- 3번은 유지보수 문제입니다
- 4번은 개발 환경 정리 문제입니다

## 2. 배너가 없어도 메인 페이지가 안전하게 열리도록 고치기

### 문제 다시 보기

현재는 배너가 비어 있으면 캐러셀이 깨질 수 있습니다.

이 문제를 고치는 가장 쉬운 방법은 아래 둘 중 하나입니다.

- 방법 A. 비어 있으면 임시 슬라이드를 보여주기
- 방법 B. 비어 있으면 캐러셀 대신 "준비 중" 문구를 보여주기

초보자에게는 **방법 B**가 더 쉽고 안전합니다.

### Step 1. 슬라이드가 비어 있는지 먼저 검사하세요

`src/js/main.js`의 `buildTrack()`를 아래처럼 바꿔보세요.

```js
function buildTrack(newSlides) {
  slides = Array.isArray(newSlides) ? newSlides : [];
  track.innerHTML = "";

  if (slides.length === 0) {
    track.innerHTML = `
      <div class="w-full py-16 text-center text-empress">
        등록된 배너가 없습니다.
      </div>
    `;
    counter.textContent = "0 / 0";
    prevLabel.textContent = "";
    nextLabel.textContent = "";
    prevBtn.disabled = true;
    nextBtn.disabled = true;
    return false;
  }

  prevBtn.disabled = false;
  nextBtn.disabled = false;

  track.append(createCarouselSlide(slides[slides.length - 1]));
  slides.forEach((slide) => track.append(createCarouselSlide(slide)));
  track.append(createCarouselSlide(slides[0]));
  return true;
}
```

여기서 중요한 포인트는 `return false`입니다.

이 뜻은:

- 슬라이드를 정상적으로 만들었으면 `true`
- 데이터가 없어서 만들지 못했으면 `false`

라는 신호를 밖으로 보내는 것입니다.

### Step 2. 이후 로직도 같이 멈추세요

슬라이드가 없는데도 `translate`, `nav`, `drag`를 계속 실행하면 또 깨질 수 있습니다.

그래서 `renderSlides()`와 초기 렌더링에서도 검사해야 합니다.

```js
function renderSlides(newSlides) {
  const built = buildTrack(newSlides);
  if (!built) return;

  currentVIdx = 1;
  applyTranslate(getTranslateForIdx(currentVIdx), false);
  updateSlideStyles();
  updateNav();
}

const built = buildTrack(slides);
if (!built) return;
```

### Step 3. 1개짜리 슬라이드도 생각해보세요

배너가 1개뿐인 경우에는 무한 캐러셀보다 정지 화면이 더 자연스러울 수 있습니다.

예를 들면:

```js
if (slides.length === 1) {
  track.append(createCarouselSlide(slides[0]));
  counter.textContent = "1 / 1";
  prevLabel.textContent = "";
  nextLabel.textContent = "";
  prevBtn.disabled = true;
  nextBtn.disabled = true;
  return false;
}
```

### 왜 이렇게 나누나요?

초보자는 종종 "데이터가 없으면 그냥 에러가 나겠지"라고 생각합니다.  
하지만 좋은 UI는 에러를 보여주는 것이 아니라, **에러가 나지 않도록 미리 막는 것**입니다.

## 3. `innerHTML` 대신 안전한 DOM 생성으로 바꾸기

### 문제 다시 보기

지금 배너 슬라이드는 문자열 HTML로 만들어집니다.

```js
el.innerHTML = `
  <figure>
    <img src="${imageUrl}" alt="${name}" />
  </figure>
`;
```

이 방식은 빠르게 만들 때는 편하지만, 외부 데이터가 들어오면 위험합니다.

### Step 1. `createElement()`로 직접 만드세요

`src/components/ui/carouselSlide.js`를 아래처럼 바꿔보세요.

```js
export function createCarouselSlide({ imageUrl, name }) {
  const el = document.createElement("div");
  el.className =
    "rotating-slide flex-shrink-0 select-none w-[min(450px,90vw)] md:w-[450px] lg:w-[914px] xl:w-[942px]";
  el.style.transition = "transform 0.45s ease, opacity 0.45s ease";

  const figure = document.createElement("figure");
  figure.className =
    "w-full aspect-[450/638] md:h-[638px] md:[aspect-ratio:auto] lg:h-[457px] xl:h-[471px] overflow-hidden shadow-sm bg-[#f4f1ea]";

  const img = document.createElement("img");
  img.src = imageUrl ?? "";
  img.alt = name ?? "";
  img.className =
    "w-full h-full object-contain md:object-cover pointer-events-none";
  img.draggable = false;

  figure.append(img);
  el.append(figure);
  return el;
}
```

이 방식의 장점:

- 텍스트는 텍스트로 들어갑니다
- HTML 태그로 해석되지 않습니다
- 나중에 버튼, 링크, 설명 문구를 추가하기 쉽습니다

### Step 2. 링크를 쓸 예정이면 주소도 검증하세요

`fetchCarousels()`에서 `href`를 함께 내려주고 있으니, 나중에 링크 버튼을 붙일 수 있습니다.

그때는 아래처럼 생각하세요.

- `javascript:` 같은 이상한 주소는 막기
- 빈 값이면 `#` 또는 버튼 숨김 처리

예시:

```js
function getSafeHref(value) {
  if (!value) return "#";
  if (value.startsWith("http://") || value.startsWith("https://") || value.startsWith("/")) {
    return value;
  }
  return "#";
}
```

### Step 3. TailwindCSS로 표현을 유지하세요

이 작업은 CSS 파일을 늘릴 필요가 없습니다.  
지금처럼 `className`에 Tailwind 유틸리티를 두고, DOM만 안전하게 만드는 쪽이 더 좋습니다.

즉:

- 구조 생성은 JavaScript
- 스타일 표현은 TailwindCSS
- 최소한의 inline style만 사용

이 조합이 지금 프로젝트 방향과 잘 맞습니다.

## 4. 카테고리 데이터를 한 곳에서 관리하도록 바꾸기

### 문제 다시 보기

현재는 상품 관리 화면에서 카테고리 ID와 이름을 직접 적고 있습니다.

```html
<option value="2">핸드 크림</option>
<option value="3">핸드 워시 & 솝</option>
```

이 방식은 처음에는 빨라 보이지만, 나중에 반드시 수정 비용이 커집니다.

### Step 1. "카테고리 가져오기"를 별도 모듈로 분리하세요

지금 `admin/js/categories.js`에는 API 함수와 화면 렌더링 함수가 같이 들어 있습니다.

이 상태는 응집도가 조금 아쉽습니다.

- `loadCategories()`는 데이터 역할
- `renderCategoriesSection()`은 화면 역할

역할이 다르기 때문입니다.

아래처럼 분리하면 더 이해하기 쉬워집니다.

```js
// admin/js/category-service.js
import { fetchAPI } from "./api.js";

export async function loadCategories() {
  return fetchAPI("/categories");
}

export function flattenCategories(categories, parentName = null) {
  const result = [];

  categories.forEach((category) => {
    result.push({ ...category, parentName });

    if (category.children?.length) {
      result.push(...flattenCategories(category.children, category.name));
    }
  });

  return result;
}
```

이렇게 하면:

- 상품 관리 화면도 이 함수를 가져다 쓸 수 있고
- 카테고리 관리 화면도 그대로 쓸 수 있고
- "카테고리를 다루는 규칙"이 한 파일에 모입니다

이게 바로 **응집도를 높이는 방법**입니다.

### Step 2. 상품 필터 드롭다운을 API 데이터로 채우세요

`admin/js/products.js`에서는 하드코딩 대신 데이터를 받아서 옵션을 만들어보세요.

```js
import { loadCategories, flattenCategories } from "/admin/js/category-service.js";

async function renderCategoryFilter() {
  const categories = await loadCategories();
  const flatCategories = flattenCategories(categories);
  const select = document.getElementById("product-category-filter");

  select.innerHTML = '<option value="">전체 카테고리</option>';

  flatCategories.forEach((category) => {
    const option = document.createElement("option");
    option.value = category.id;
    option.textContent = category.parentName
      ? `${category.parentName} > ${category.name}`
      : category.name;
    select.append(option);
  });
}
```

### Step 3. 상품 등록 폼도 숫자 입력 대신 `select`로 바꾸세요

현재는 상품 등록/수정 폼에서 카테고리를 숫자로 직접 입력합니다.

```html
<input type="number" id="p-category" class="form-input" />
```

초보자 입장에서는 아주 실수하기 쉬운 입력 방식입니다.

아래처럼 바꾸는 편이 훨씬 낫습니다.

```html
<select id="p-category" class="form-input"></select>
```

그리고 같은 카테고리 데이터를 재사용하세요.

이렇게 하면:

- 잘못된 숫자 입력을 막을 수 있고
- 필터와 등록 폼이 같은 기준을 쓰게 되고
- 카테고리가 추가되어도 자동으로 반영됩니다

### Step 4. 가능하면 TailwindCSS로 옮길 준비를 하세요

지금 관리자 페이지는 기존 공통 클래스(`search-input`, `form-input`) 중심입니다.  
한 번에 다 바꾸기는 어렵더라도, 새 UI를 만들 때는 아래 원칙을 추천합니다.

- 새 전용 CSS를 추가하기보다 Tailwind 유틸리티로 레이아웃을 표현하기
- 정말 여러 곳에서 반복되는 스타일만 공통 클래스로 묶기
- "데이터 렌더링 로직"과 "스타일 규칙"을 한 문자열에 너무 많이 섞지 않기

즉, CSS를 늘리기보다 Tailwind를 우선하고, 기존 CSS는 최소한만 유지하는 방향이 좋습니다.

## 5. proxy 설정은 실제 호출 방식과 맞춰서 정리하기

### 문제 다시 보기

지금은 Vite proxy를 추가했지만, API 호출은 절대 URL을 사용하고 있습니다.

그래서 proxy가 실제로는 동작하지 않습니다.

### 방법 A. proxy를 쓸 계획이면 상대 경로로 바꾸세요

예시:

```js
const BASE_URL = import.meta.env.DEV
  ? "/api/loccishop/v1"
  : "https://api.fullstackfamily.com/api/loccishop/v1";
```

이렇게 하면 개발 환경에서는 proxy를 타고, 배포 환경에서는 실제 주소를 사용할 수 있습니다.

### 방법 B. 아직 안 쓸 거라면 proxy를 지우세요

초보 팀에서는 "미리 넣어 둔 설정"이 오히려 헷갈림을 만들 때가 많습니다.

그래서 아직 상대 경로 구조로 바꿀 계획이 없다면:

- proxy는 잠시 제거하고
- 나중에 환경 변수 구조를 잡을 때 다시 넣는 편이 낫습니다

### 추천

지금 단계에서는 **방법 A**가 가장 좋습니다.  
왜냐하면:

- 로컬 개발과 배포 환경을 분리해서 생각하는 연습이 되고
- 주소가 바뀌어도 한 곳만 수정하면 되고
- 백엔드 주소에 대한 결합도도 낮아지기 때문입니다

## 6. 마무리

이번 수정에서 가장 먼저 고칠 것은 아래 두 가지입니다.

1. 배너가 비어 있을 때도 메인 페이지가 깨지지 않게 만들기
2. `innerHTML` 대신 안전한 DOM 생성으로 바꾸기

그 다음에 구조를 더 좋게 만들고 싶다면:

3. 카테고리 데이터를 한 곳에서 관리하기
4. proxy와 API 주소 체계를 정리하기

좋은 코드는 "짧은 코드"가 아니라, **예외 상황에서도 덜 깨지는 코드**입니다.  
그리고 좋은 구조는 "많이 나눈 구조"가 아니라, **각 파일이 자기 역할만 아는 구조**입니다.

## 0. 이번 리뷰 범위

이번 리뷰는 아래 마이페이지 관련 변경 파일을 중심으로 진행했습니다.

- `src/pages/mypage/index.html`
- `src/pages/mypage/index.js`
- `src/pages/mypage/components/my-info.html`
- `src/pages/mypage/components/my-order.html`
- `src/pages/mypage/components/my-wish.html`
- `src/pages/mypage/components/my-review.html`
- `src/pages/mypage/handlers/initMypageEvents.js`
- `src/pages/mypage/handlers/renderInfo.js`
- `src/pages/mypage/handlers/renderOrderList.js`
- `src/pages/mypage/handlers/renderWishList.js`
- `src/pages/mypage/handlers/renderReviewList.js`

아래 두 파일도 함께 확인했습니다. 이유는 새로 추가된 마이페이지가 실제 서비스에 연결되는지 확인하려면 같이 봐야 하기 때문입니다.

- `vite.config.js`
- `src/components/header/header.js`

리뷰 기준은 아래와 같습니다.

- 초보 개발자도 읽고 수정할 수 있는 구조인가
- 한 파일이 한 가지 역할에 집중하고 있는가
- 다른 파일, 경로, 문자열에 너무 강하게 묶여 있지 않은가
- CSS를 최소화하고 TailwindCSS 중심으로 작성할 수 있는가
- 데스크톱과 모바일에서 모두 안전하게 열리는가

---

## 1. 먼저, 잘한 점

### 섹션을 나누려는 방향은 좋았습니다

`my-info`, `my-order`, `my-wish`, `my-review`로 마이페이지 하위 화면을 분리한 것은 좋은 시작입니다.

이 방식의 장점은:

- 파일 이름만 봐도 어떤 화면인지 알 수 있습니다
- 한 번에 큰 HTML을 읽지 않아도 됩니다
- 나중에 각 섹션 담당자를 나눠 협업하기 쉬워집니다

즉, "큰 페이지를 섹션 단위로 나누자"는 방향 자체는 맞습니다.

### 데이터 바인딩을 위한 `id`를 미리 넣어둔 점이 좋습니다

예를 들면 아래와 같은 부분입니다.

- `info-summary-name`
- `info-summary-email`
- `status-shipping`
- `wish-count`
- `review-list-container`

이런 식으로 "나중에 JavaScript가 값을 넣을 위치"를 미리 정리해둔 것은 초보 단계에서 매우 좋은 습관입니다.

### 새 CSS 파일을 만들지 않고 Tailwind 유틸리티로 표현한 점이 좋습니다

이번 작업은 별도 CSS 파일을 늘리지 않고 대부분 Tailwind 클래스 안에서 해결하고 있습니다.

프로젝트 방향이 "CSS는 꼭 필요할 때만 최소화"라면, 이 선택은 좋은 방향입니다.

## 2. 개선이 필요한 점

### 2-1. 배포 시 마이페이지가 열리지 않을 가능성이 큽니다 (중요도: 높음)

**파일**

- `src/components/header/header.js:42-45`
- `src/pages/mypage/handlers/renderInfo.js:6-10`
- `src/pages/mypage/handlers/renderOrderList.js:5-12`
- `src/pages/mypage/handlers/renderWishList.js:6-14`
- `src/pages/mypage/handlers/renderReviewList.js:6-14`
- `vite.config.js:21-37`

현재 구조는 개발 서버에서는 얼핏 동작할 수 있지만, 배포 빌드 기준으로는 매우 불안합니다.

이유는 두 가지입니다.

첫째, `vite.config.js`의 `build.rollupOptions.input`에 `mypage` 엔트리가 없습니다.

즉, 로그인 후 헤더에서 아래 경로로 이동하더라도:

```js
window.location.href = "/src/pages/mypage/index.html";
```

빌드 결과물 안에 이 페이지가 포함되지 않을 수 있습니다.

둘째, 각 섹션을 아래처럼 `fetch("/src/pages/...html")`로 다시 읽고 있습니다.

```js
const response = await fetch("/src/pages/mypage/components/my-info.html");
```

이 방식은 화면 로직이 "소스 폴더 구조"에 직접 묶여 있다는 뜻입니다.

- 폴더가 조금만 바뀌어도 깨집니다
- 번들링 결과가 달라지면 깨집니다
- 로컬 개발 경로와 배포 경로가 달라지면 깨집니다

이런 구조는 **결합도가 높은 구조**입니다.

좋은 구조는 "이 파일이 `src/pages/...`에 있는지"를 화면 코드가 몰라도 동작해야 합니다.

#### 왜 초보 개발자에게 특히 중요한가요?

처음에는 "보이니까 괜찮다"라고 생각하기 쉽습니다.  
하지만 프론트엔드에서는 **개발 서버에서 보이는 것**과 **배포에서 실제로 열리는 것**이 다를 수 있습니다.

이번 코드는 바로 그 위험이 있습니다.

#### 참고

이번 턴에서 `npm install` 후 `npm run build`로 확인하려 했지만, 현재 저장소는 별도 기존 이슈 때문에 먼저 빌드가 실패했습니다.

- 빌드 실패 원인: `vite.config.js`가 `src/pages/signup/index.html`을 엔트리로 잡고 있는데 실제 파일은 없습니다

즉, 마이페이지 검증 전에 저장소 전체 빌드 설정부터 이미 막혀 있었습니다.  
하지만 그와 별개로, `mypage` 엔트리 누락과 `fetch("/src/pages/...")` 구조 자체는 코드상에서 이미 확인되는 문제입니다.

### 2-2. 실제 사용자 화면에 가짜 개인정보와 가짜 통계가 그대로 보입니다 (중요도: 높음)

**파일**

- `src/pages/mypage/components/my-info.html:6-44`
- `src/pages/mypage/components/my-order.html:1-28`
- `src/pages/mypage/components/my-wish.html:4-8`
- `src/pages/mypage/components/my-review.html:7-18`

현재 마크업에는 아래 값들이 실제 텍스트로 들어 있습니다.

- `홍길동님`
- `hong@naver.com`
- `가입일 : 2026-04-16`
- `15,000P`
- 주문 상태 수치 `0`, `1`, `2`
- 위시리스트 `3 items`
- 리뷰 평점 `4.7`
- `Based on 3 reviews`

문제는 이 값들이 "예시"인지 "실제 데이터"인지 화면에서 구분되지 않는다는 점입니다.

사용자 입장에서는:

- 내가 가입한 날짜가 진짜 2026-04-16인지
- 내 포인트가 실제로 15,000P인지
- 위시리스트가 정말 3개인지

구분할 수 없습니다.

즉, 지금은 **레이아웃용 샘플 데이터**와 **실제 서비스 데이터**가 한 파일 안에 섞여 있습니다.

이 구조는 응집도도 떨어지고, 나중에 데이터 바인딩할 때도 헷갈리게 만듭니다.

초보 개발자는 여기서 이렇게 기억하면 좋습니다.

> 화면 뼈대와 실제 데이터는 분리해야 합니다.  
> 뼈대는 비어 있어도 되고, 데이터는 나중에 넣으면 됩니다.

### 2-3. 사이드바 활성화 상태가 HTML과 JavaScript에 나뉘어 있어서 클릭 후 스타일이 꼬일 수 있습니다 (중요도: 중간)

**파일**

- `src/pages/mypage/index.html:17-76`
- `src/pages/mypage/handlers/initMypageEvents.js:37-47`

현재 사이드바는 처음 HTML에서 이미 "내 정보만 활성화된 모양"을 직접 써두었습니다.

반면 클릭 후에는 JavaScript가 일부 클래스만 더하고 뺍니다.

```js
btn.classList.remove("bg-merino", "text-ferra", "font-semibold");
btn.classList.add("text-empress");
```

이 방식의 문제는 "활성/비활성 상태"를 한 곳에서 관리하지 않는다는 점입니다.

예를 들면:

- 처음 활성 버튼은 `font-bold`인데, 비활성화될 때 이 클래스는 제거되지 않습니다
- 비활성 버튼이 원래 갖고 있던 `hover` 관련 클래스와 경계선 스타일은 상태 변경 후 일관되지 않을 수 있습니다
- 새로 활성화된 버튼은 비활성 클래스와 활성 클래스가 같이 남을 수 있습니다

즉, 상태가 바뀔수록 스타일 기준이 점점 헷갈려집니다.

이것은 작은 문제처럼 보여도, 메뉴가 늘어나면 바로 관리가 어려워집니다.

이유는 간단합니다.

- HTML도 상태를 알고 있고
- JavaScript도 상태를 알고 있고
- 두 곳이 조금만 달라도 UI가 어긋납니다

이건 **결합도가 높은 구조**입니다.

### 2-4. 모바일 레이아웃이 고려되지 않아 작은 화면에서 바로 깨질 가능성이 큽니다 (중요도: 중간)

**파일**

- `src/pages/mypage/index.html:13-15`
- `src/pages/mypage/components/my-info.html:64-103`
- `src/pages/mypage/components/my-order.html:2-45`
- `src/pages/mypage/components/my-wish.html:12`

현재 레이아웃은 대부분 데스크톱 기준으로 고정되어 있습니다.

예를 들면:

- 바깥 레이아웃이 `flex` 한 줄 배치입니다
- 사이드바가 `w-56` 고정입니다
- 개인정보 영역이 `grid-cols-2` 고정입니다
- 위시리스트가 `grid-cols-3` 고정입니다
- 주문 요약은 가로 4칸 구조를 바로 사용하고 있습니다

이 상태에서 모바일 폭으로 줄어들면:

- 사이드바와 본문이 가로로 눌려서 읽기 어려워지고
- 카드와 텍스트가 너무 좁아지고
- 테이블 역할을 하는 줄이 겹치거나 넘칠 수 있습니다

이번 프로젝트는 TailwindCSS를 사용하고 있으니, 이런 문제는 별도 CSS 파일 없이도 충분히 해결할 수 있습니다.

예를 들어:

- `flex-col lg:flex-row`
- `grid-cols-1 md:grid-cols-2`
- `grid-cols-1 sm:grid-cols-2 xl:grid-cols-3`
- `overflow-x-auto`

처럼 반응형 클래스를 쓰면 됩니다.

즉, 이 문제는 CSS를 더 만드는 방향이 아니라 **Tailwind의 반응형 유틸리티를 더 잘 쓰는 방향**으로 해결해야 합니다.

### 2-5. 같은 렌더링 코드가 4번 반복되고, 에러 처리도 서로 다릅니다 (중요도: 중간)

**파일**

- `src/pages/mypage/handlers/renderInfo.js:1-15`
- `src/pages/mypage/handlers/renderOrderList.js:1-16`
- `src/pages/mypage/handlers/renderWishList.js:1-18`
- `src/pages/mypage/handlers/renderReviewList.js:1-18`
- `src/pages/mypage/index.js:4-11`
- `src/pages/mypage/handlers/initMypageEvents.js:20-32`

현재 네 개의 렌더 함수는 거의 같은 일을 합니다.

1. `#mypage-content`를 찾고
2. HTML 파일을 `fetch`하고
3. `innerHTML`로 넣고
4. 실패하면 에러를 처리합니다

그런데 구현은 서로 조금씩 다릅니다.

- `renderInfo()`는 화면에 오류 문구를 보여줍니다
- 다른 세 함수는 콘솔에만 찍고 끝납니다
- 어떤 함수는 메시지가 한국어이고, 어떤 함수는 `HTTP error!` 형식입니다

이렇게 되면 나중에 수정할 때 네 파일을 모두 고쳐야 합니다.

예를 들어:

- 로딩 UI를 추가하고 싶을 때
- 오류 UI를 통일하고 싶을 때
- 렌더 방식 자체를 바꾸고 싶을 때

같은 수정을 여러 군데 반복하게 됩니다.

이건 초보 개발자가 가장 빨리 줄여야 하는 중복입니다.

중복은 단순히 "줄 수가 많다"는 문제가 아니라,  
**한 군데 고쳤는데 다른 곳은 안 고쳐서 버그가 생기는 문제**로 이어집니다.

## 3. 우선순위 정리

초보 개발자라면 아래 순서로 보는 것을 추천합니다.

1. 배포 경로와 빌드 연결 문제부터 해결하기
2. 샘플 텍스트와 실제 데이터 영역을 분리하기
3. 메뉴 활성 상태를 한 곳에서 관리하도록 바꾸기
4. 모바일 반응형을 Tailwind로 정리하기
5. 중복된 렌더 함수를 공통 함수로 합치기

## 4. 한 줄 총평

이번 마이페이지 작업은 "큰 화면을 섹션 단위로 나누는 방향"은 좋았습니다.  
다만 지금 상태는 **레이아웃은 예쁘게 보이지만, 실제 서비스 구조로 연결되는 부분이 약하고, 데이터/상태/반응형이 아직 분리되지 않은 상태**입니다.

즉, 지금은 "디자인 초안"으로는 괜찮지만, "운영 가능한 페이지"로 가려면 한 단계 더 구조를 다듬어야 합니다.

## 2. 잘한 점

### API 계층 분리가 교과서적이에요!

```
src/js/api/
├── client.js          ← 공통 fetch 래퍼 (에러 처리 포함)
└── product/
    └── index.js       ← 상품 관련 API 함수들
```

```js
// client.js — 한 곳에서 에러 처리!
export async function fetchAPI(path) {
  const res = await fetch(`${BASE_URL}${path}`);
  if (!res.ok) throw new Error(`API 오류: ${res.status}`);
  const json = await res.json();
  if (!json.success) throw new Error("서버 오류가 발생했습니다.");
  return json.data;
}

// product/index.js — 이 함수를 사용하는 곳은 에러 처리를 신경 쓸 필요 없어요!
export function fetchProduct(id) {
  return fetchAPI(`/products/${id}`);
}
```

이것이 바로 **관심사의 분리(Separation of Concerns)** 패턴이에요! API 호출이라는 하나의 관심사를 `api/` 폴더로 분리했기 때문에:
- 나중에 API 주소가 바뀌면 `client.js` **한 곳만** 수정하면 돼요
- 에러 처리도 한 곳에서 관리하니까 일관적이에요
- `fetchProduct(id)`처럼 **이름만 봐도 뭘 하는지 알 수 있는** 함수가 됐어요

정말 훌륭한 설계예요!

### handlers/ 디렉토리 분리가 인상적이에요!

```
handlers/
├── renderProductMain.js      ← 상품 메인 영역 렌더링
```

이전 리뷰에서 `index.js`에서 함수를 분리한 것도 좋았는데, 이번에는 아예 **파일 단위로 분리**했어요! 각 파일이 하나의 기능만 담당하고 있어요. 이것이 바로 **높은 응집도(High Cohesion)**의 완성형이에요!

특히 `renderStars.js`를 별도 유틸로 뽑아낸 것이 좋아요. 별점 렌더링은 베스트 리뷰, 전체 리뷰, 상품 메인 등 **여러 곳에서 필요**하니까요.

## 3. 개선하면 좋을 점

### 3-1. 리뷰 목록에서 `innerHTML`에 사용자 데이터를 직접 넣으면 위험해요 — XSS 공격 (중요도: 높음)

**현재 상태:**

```js
// src/pages/product/detail/handlers/initReviews.js 46~71번째 줄
reviews.forEach((review) => {
  const li = document.createElement("li");
  li.innerHTML = `
    <article class="...">
      <h3 class="review-title leading-6">${review.title ?? ""}</h3>
      <div class="flex justify-between text-xs font-medium">
        <span>${review.author}</span>
        <span>${review.createdAt.slice(0, 10)}</span>
      </div>
      <p class="text-sm leading-5 text-zambezi">${review.content}</p>
    </article>
  `;
  reviewList.append(li);
});
```

그런데 같은 프로젝트의 `initBestReview.js`에서는 **안전하게** 처리하고 있어요:

```js
// src/pages/product/detail/handlers/initBestReview.js 23~25번째 줄
document.querySelector("#review-title").textContent = review.title ?? "";
document.querySelector("#review-content").textContent = review.content;
document.querySelector("#author").textContent = review.author;
```

**왜 개선하면 좋을까?**

`innerHTML`에 사용자 데이터를 직접 넣으면 **XSS(Cross-Site Scripting) 공격**에 노출돼요. 쉽게 말하면, 누군가 리뷰 제목에 이런 내용을 작성하면:

```
훌륭한 제품<img src=x onerror="alert('해킹!')">
```

이 코드가 `innerHTML`을 통해 **실제 HTML로 실행**돼요! 공격자가 다른 사용자의 쿠키를 훔치거나, 가짜 로그인 폼을 띄울 수 있어요.

반면 `textContent`는 HTML 태그를 **그냥 텍스트로 취급**해서 안전해요. `initBestReview.js`에서 이미 이 방법을 쓰고 있어요!

> **OWASP Top 10**(웹 보안 위협 목록)에서 XSS는 항상 상위권에 있는 대표적인 보안 취약점이에요.

**추천:**

`innerHTML` 대신 **DOM 요소를 직접 생성**하는 방식으로 바꾸세요:

```js
// 수정 후 — DOM 요소를 직접 생성하여 안전하게!
reviews.forEach((review) => {
  const li = document.createElement("li");
  li.className = "h-full";

  const article = document.createElement("article");
  article.className = "h-full flex flex-col gap-3 shadow-[0_0_6px_0_rgba(0,0,0,0.05)] rounded-lg p-5 bg-rose-white";

  const title = document.createElement("h3");
  title.className = "review-title leading-6";
  title.textContent = review.title ?? "";  // ← textContent = 안전!

  const content = document.createElement("p");
  content.className = "text-sm leading-5 text-zambezi";
  content.textContent = review.content;     // ← textContent = 안전!

  article.append(title, content);
  li.append(article);
  reviewList.append(li);
});
```

코드가 좀 더 길어지지만, **보안은 코드 길이보다 중요**해요!

**왜 개선하면 좋을까?**

**문제 1: `<header>` 중첩**
`renderHeader()`가 `<header>` 요소를 만들어서 반환하고, `main.js`에서 이걸 `<header id="header">` 안에 넣어요. 결과적으로 `<header>` 안에 또 `<header>`가 생겨요. HTML에서 `<header>`를 중첩하는 것은 **시맨틱적으로 올바르지 않아요**. 스크린리더가 "헤더 시작... 헤더 시작..."이라고 두 번 읽어요.

**문제 2: `<ul>` 안에 `<li>` 없이 `<img>` 직접 사용**
HTML 표준에서 `<ul>`의 자식 요소는 **반드시 `<li>`**여야 해요. `<img>`를 직접 넣으면 W3C 유효성 검사에서 오류가 나요.

**문제 3: 모든 이미지에 `alt=""` (빈 대체 텍스트)**
로고 이미지의 `alt`가 비어 있으면 스크린리더가 "이미지"라고만 읽고 넘어가요. 사용자가 어떤 사이트인지 알 수 없어요.

**추천:**

```js
// 수정 후
export function renderHeader() {
  const nav = document.createElement("nav");    // ← <header> 대신 <nav> 사용!
  nav.className = "bg-spring-wood shadow p-4 w-full h-16";

  nav.innerHTML = `
    <div class="h-full flex items-center justify-between max-w-sm mx-auto">
      <button type="button" class="hamburger-btn cursor-pointer">
        <img src="/src/assets/icon/Hamburger.svg" alt="메뉴 열기" />
      </button>
      <a href="/">
        <img src="/src/assets/logo/Loccitane.svg" alt="L'Occitane 홈으로" />
      </a>
      <ul class="flex gap-4">
        <li><a href="#"><img src="/src/assets/icon/myPage.svg" alt="마이페이지" /></a></li>
        <li><a href="#"><img src="/src/assets/icon/heart-empty.svg" alt="위시리스트" /></a></li>
        <li><a href="#"><img src="/src/assets/icon/cart.svg" alt="장바구니" /></a></li>
      </ul>
    </div>
  `;

  return nav;
}
```

변경 포인트:
- `document.createElement("header")` → `"nav"`: 외부 `<header>` 안에 `<nav>`이 들어가므로 중첩 해소
- 햄버거 아이콘을 `<button>`으로 감쌈: 키보드로도 접근 가능
- `<ul>` 안에 `<li>` 추가: HTML 표준 준수
- `alt` 속성에 의미 있는 텍스트 추가: 접근성 향상

### 3-3. `bg-header-footer` 색상이 정의되어 있지 않아요 (중요도: 높음)

**현재 상태:**

```js
// src/components/header/header.js 3번째 줄
header.className = "bg-header-footer shadow p-4 w-full h-16";

// src/components/footer/footer.js 4번째 줄
footer.className = "w-full bg-header-footer pt-10 pb-8 border-t border-gray-200";
```

하지만 `src/styles/style.css`의 `@theme` 블록에는:

```css
/* style.css — 정의된 색상들 */
--color-rose-white: #fffefe;
--color-spring-wood: #fbf9f6;
--color-merino: #f7f2ea;
/* ... */
/* --color-header-footer 는 어디에도 없어요! */
```

**왜 개선하면 좋을까?**

Tailwind CSS v4에서 `bg-header-footer`라는 클래스를 쓰면, 내부적으로 `background-color: var(--color-header-footer)`를 생성해요. 하지만 `--color-header-footer`가 정의되지 않았으니 이 CSS 변수의 값은 **비어 있어요**. 결과적으로 **헤더와 푸터에 배경색이 적용되지 않아요**!

개발자 도구(F12)에서 확인하면 `background-color`가 빈 값인 것을 볼 수 있어요.

**추천:**

두 가지 방법 중 하나를 선택하세요:

**방법 1: style.css에 색상 추가**

```css
/* style.css @theme 블록에 추가 */
--color-header-footer: #fbf9f6;    /* spring-wood와 같은 색 또는 디자인 시안에 맞는 색 */
```

**방법 2: 이미 정의된 색상 사용 (추천!)**

```js
// header.js — 이미 있는 색상 활용
header.className = "bg-spring-wood shadow p-4 w-full h-16";

// footer.js
footer.className = "w-full bg-spring-wood pt-10 pb-8 border-t border-gray-200";
```

> **쉽게 말하면:** 물감 상자에 "하늘색"이 없는데 "하늘색으로 칠해"라고 적어둔 것과 같아요. 물감 상자에 있는 색 이름을 써야 해요!

### 3-4. 상품 목록 페이지에 헤더/푸터가 나타나지 않아요 (중요도: 높음)

**현재 상태:**

```html
<!-- src/pages/product/list/index.html -->
<body>
  <main>
    <section class="relative w-full h-119 ...">
      <!-- 히어로 배너 -->
    </section>
    <!-- ... 콘텐츠 ... -->
  </main>
</body>
```

다른 페이지들은 헤더/푸터 컨테이너가 있어요:

```html
<!-- index.html -->
<header id="header"></header>
<main>...</main>
<footer id="footer"></footer>

<!-- src/pages/product/detail/index.html -->
<header id="header"></header>
<main>...</main>
<footer id="footer"></footer>
```

**왜 개선하면 좋을까?**

`main.js`에서 헤더와 푸터를 렌더링하는 코드가 이렇게 동작해요:

```js
// main.js
const headerContainer = document.getElementById("header");
const footerContainer = document.getElementById("footer");
if (headerContainer) headerContainer.append(renderHeader());
if (footerContainer) footerContainer.append(renderFooter());
```

`document.getElementById("header")`가 **`null`을 반환**하기 때문에 `if (headerContainer)` 조건을 통과하지 못해요. 에러는 나지 않지만, **헤더와 푸터가 아예 표시되지 않아요!**

상품 목록 페이지만 헤더/푸터 없이 콘텐츠만 나타나는 상태예요.

**추천:**

```html
<!-- 수정 전 (src/pages/product/list/index.html) -->
<body>
  <main>

<!-- 수정 후 — header/footer 컨테이너 추가 -->
<body>
  <header id="header"></header>
  <main>
    <!-- 기존 콘텐츠 그대로 -->
  </main>
  <footer id="footer"></footer>
</body>
```

> **쉽게 말하면:** 다른 방에는 전등 소켓이 있어서 전등이 켜지는데, 이 방만 소켓이 없어서 전등을 꽂을 곳이 없는 거예요!

### 3-5. 햄버거 메뉴 아이콘이 `<img>`라서 키보드로 접근할 수 없어요 (중요도: 중간)

**현재 상태:**

```js
// src/components/header/header.js 7번째 줄
<img src="/src/assets/icon/Hamburger.svg" alt="" class="hamburger-btn cursor-pointer" />
```

```js
// src/js/main.js 19~23번째 줄
const hamburgerBtn = document.querySelector(".hamburger-btn");
if (hamburgerBtn) {
  hamburgerBtn.addEventListener("click", () => {
    menuDrawer.open();
  });
}
```

**왜 개선하면 좋을까?**

`<img>` 태그는 **인터랙티브 요소가 아니에요**. 즉:
- 키보드의 **Tab 키로 이동**할 수 없어요 (포커스를 받지 못해요)
- 키보드의 **Enter 키로 클릭**할 수 없어요
- 스크린리더가 **"버튼"이라고 읽지 않아요**

시각적으로는 `cursor-pointer`가 있어서 마우스로 클릭할 수 있지만, **키보드만 사용하는 사용자**는 메뉴를 열 수 없어요.

**추천:**

```js
// 수정 전
<img src="/src/assets/icon/Hamburger.svg" alt="" class="hamburger-btn cursor-pointer" />

// 수정 후 — <button>으로 감싸기
<button type="button" class="hamburger-btn cursor-pointer" aria-label="메뉴 열기">
  <img src="/src/assets/icon/Hamburger.svg" alt="" aria-hidden="true" />
</button>
```

- `<button>`: Tab키로 이동 가능, Enter키로 클릭 가능
- `aria-label="메뉴 열기"`: 스크린리더가 "메뉴 열기 버튼"이라고 읽어줘요
- `alt="" aria-hidden="true"`: 아이콘 이미지 자체는 장식이니까 스크린리더가 무시

> **쉽게 말하면:** 엘리베이터 버튼이 **그림**으로만 그려져 있으면 눌러도 반응이 없어요. 진짜 **버튼**이어야 눌릴 수 있어요!

### 3-6. `initReviews`에서 리뷰를 999개 한꺼번에 불러와요 (중요도: 중간)

**현재 상태:**

```js
// src/pages/product/detail/handlers/initReviews.js 4~9번째 줄
export async function initReviews(productId, { sort = "latest" } = {}) {
  const { reviews, meta } = await fetchProductReviews(productId, {
    page: 1,
    limit: 999,    // ← 999개를 한 번에!
    sort,
  });
```

**왜 개선하면 좋을까?**

한 번에 999개의 리뷰를 불러오면:
- **네트워크 비용**: 데이터가 많으면 로딩이 느려져요. 모바일 사용자는 특히 답답할 수 있어요
- **메모리 사용**: 999개의 리뷰 DOM 요소를 한 번에 생성하면 스크롤이 버벅일 수 있어요
- **불필요한 데이터**: 사용자가 실제로 읽는 리뷰는 보통 첫 페이지 10~20개 정도예요

참고로 `meta.pagination`에 페이지네이션 정보가 이미 있어서, API가 페이지네이션을 지원하고 있어요!

**추천:**

```js
// 수정 전
const { reviews, meta } = await fetchProductReviews(productId, {
  page: 1,
  limit: 999,
  sort,
});

// 수정 후 — 적절한 양만 불러오기
const { reviews, meta } = await fetchProductReviews(productId, {
  page: 1,
  limit: 10,    // ← 첫 페이지 10개만
  sort,
});
```

"리뷰 더보기" 버튼이 이미 있으니까, 다음 페이지를 불러오는 기능을 추가하면 완성이에요!

> **쉽게 말하면:** 편의점에서 음료를 999개 주문하는 대신, 10개씩 필요할 때마다 추가 주문하는 것이 훨씬 효율적이에요!

### 3-8. `product-card-list.js`에서 `resize` 이벤트 리스너가 계속 쌓여요 (중요도: 중간)

**현재 상태:**

```js
// src/components/ui/product-card-list.js 92~93번째 줄
window.addEventListener("resize", updateCursor);
window.addEventListener("resize", updateCardWidths);
```

**왜 개선하면 좋을까?**

`renderProductCards` 함수가 호출될 때마다 `window`에 새로운 `resize` 이벤트 리스너가 추가돼요. 이 함수가 5번 호출되면 같은 함수가 **5번** 등록되어서 화면 크기가 바뀔 때 **같은 작업을 5번** 실행해요.

이것을 **메모리 누수(Memory Leak)**라고 해요. 특히 SPA(Single Page Application)처럼 페이지를 새로고침하지 않고 여러 번 렌더링하는 경우 문제가 돼요.

**추천:**

```js
// 수정 전 — 호출할 때마다 리스너 추가
window.addEventListener("resize", updateCursor);
window.addEventListener("resize", updateCardWidths);

// 수정 후 — 기존 리스너를 제거한 후 새로 추가
// 함수를 변수에 저장하여 제거 가능하게!
let prevCursorHandler = null;
let prevWidthHandler = null;

export function renderProductCards(products) {
  // 기존 리스너 제거
  if (prevCursorHandler) window.removeEventListener("resize", prevCursorHandler);
  if (prevWidthHandler) window.removeEventListener("resize", prevWidthHandler);

  // ... (기존 코드) ...

  // 새 리스너 등록
  prevCursorHandler = updateCursor;
  prevWidthHandler = updateCardWidths;
  window.addEventListener("resize", updateCursor);
  window.addEventListener("resize", updateCardWidths);
}
```

> **쉽게 말하면:** 알람을 하나만 맞춰야 하는데, 버튼을 누를 때마다 알람이 추가로 설정되면 아침에 10개의 알람이 한꺼번에 울리는 것과 같아요!

### 3-9. CSS에서 Tailwind로 대체할 수 있는 커스텀 스타일이 있어요 (중요도: 낮음)

**현재 상태:**

```js
// src/components/ui/product-card-list.js 22~25번째 줄
if (container.scrollWidth > container.clientWidth) {
  container.style.cursor = "grab";
} else {
  container.style.cursor = "default";
}
```

```js
// 38번째 줄
container.style.cursor = "grabbing";

// 79번째 줄
card.style.width = `${cardWidth}px`;
card.style.flexShrink = "0";
```

**왜 개선하면 좋을까?**

Tailwind CSS를 사용하는 프로젝트에서는 `style` 속성(인라인 스타일) 대신 **Tailwind 클래스**를 사용하는 것이 일관성 있어요. `cursor-grab`, `cursor-grabbing` 같은 Tailwind 유틸리티가 이미 있어요.

인라인 스타일은 Tailwind 클래스보다 **우선순위가 높아서** 나중에 Tailwind로 스타일을 바꾸려 해도 덮어쓰기가 어려워요.

다만, `card.style.width`처럼 **동적으로 계산된 값**은 JavaScript에서 설정해야 하므로 인라인 스타일이 맞아요. 문제는 `cursor` 같은 **정적인 값**을 인라인으로 넣는 것이에요.

**추천:**

```js
// 수정 전
container.style.cursor = "grab";

// 수정 후 — Tailwind 클래스 사용
container.classList.add("cursor-grab");

// 수정 전
container.style.cursor = "grabbing";

// 수정 후
container.classList.remove("cursor-grab");
container.classList.add("cursor-grabbing");
```

3. **`header.js`에서 `<header>` → `<nav>` 변경, `<ul>` 안에 `<li>` 추가** — HTML 구조 수정 (5분)
4. **`initReviews.js`에서 `innerHTML` → DOM 요소 생성** — XSS 보안 (15분)
5. **`initReviews.js`에서 `limit: 999` → `limit: 10`** — 성능 개선 (1분)
6. **`renderProductMain.js`에서 `innerHTML` → DOM 요소 생성** — 보안 습관 (10분)
7. **`product-card-list.js` resize 리스너 정리** — 메모리 누수 방지 (5분)
8. **`cursor` 인라인 스타일 → Tailwind 클래스** — 일관성 (3분)
9. **회원가입 에러 아이콘 `hidden` 추가** — 2차 리뷰 미해결 (10분)

## 6. 마무리

이번 변경에서 가장 인상적인 점은 **아키텍처 수준의 성장**이에요!

- **API 계층 분리** (`api/client.js`, `api/product/index.js`) — 서버와 통신하는 코드를 한 곳에 모았어요
- **핸들러 파일 분리** (`handlers/` 디렉토리) — 기능별로 파일을 나눠서 응집도를 높였어요
- **공통 컴포넌트** (`header.js`, `footer.js`, `product-card-list.js`) — 재사용 가능한 구조를 만들었어요
- **이전 리뷰 미해결 항목 대거 해결** — 1차 리뷰부터 남아 있던 기술 부채까지 정리했어요

이것들은 초보 단계를 넘어 **중급 수준의 코드 구조화 능력**을 보여주는 것이에요.

이번 리뷰에서 지적한 항목 중 1~2번(색상 정의, 헤더/푸터 컨테이너)은 **1분이면 해결**할 수 있는 간단한 수정이에요. 먼저 이것들부터 빠르게 고치면 눈에 보이는 문제가 바로 사라져요. 그 다음에 XSS 보안 수정을 하면 실력도 크게 올라갈 거예요.

프로젝트가 빈 HTML에서 시작해서, 지금은 API 연동까지 갖춘 **완성도 높은 쇼핑몰**이 되어가고 있어요. 코드를 읽으면 팀원들이 함께 열심히 만들고 있다는 것이 느껴져요. 이 속도면 정말 멋진 결과물이 나올 거예요. 파이팅!

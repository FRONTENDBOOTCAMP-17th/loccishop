## 2. 잘한 점

### index.js의 함수 분리가 교과서적이에요!

이전 리뷰에서 "역할별로 분리하면 좋겠어요"라고 제안했는데, **정확히 그 방향으로 개선**했어요!

```js
// 이전 코드: 모든 것이 initProductPage() 하나에
// 현재 코드: 깔끔하게 분리!
function initBadge() { /* 배지만 */ }
function initCartButton() { /* 장바구니만 */ }
function initDrawers() { /* 드로어만 */ }
function initProductCard() { /* 상품 카드만 */ }
function initMoreReviewButton() { /* 더보기 버튼만 */ }

async function initProductPage() {
  await loadHTML("#detail-main", "...");
  initBadge();           // ← 한눈에 뭘 하는지 보여요!
  initCartButton();
  // ...
}
```

각 함수가 **하나의 역할에만 집중**하고 있어요. 이것이 바로 **높은 응집도(High Cohesion)**예요! 버그가 생기면 해당 함수만 보면 되니까 디버깅도 쉬워져요. 이전 리뷰의 피드백을 빠르게 적용한 것이 정말 훌륭해요!

### product-card.js의 가로형 레이아웃 추가가 인상적이에요!

```js
export function createProductCard({
  layout = "vertical", //"vertical" | "horizontal"
} = {}) {
  if (layout === "horizontal") {
    // 가로형 카드 (리추얼 스텝용)
  }
  // 세로형 카드 (상품 목록용)
}
```

하나의 컴포넌트에서 `layout` 옵션으로 두 가지 형태를 지원하는 설계가 좋아요. **같은 데이터(상품 정보)**를 다른 형태로 보여줄 수 있어서 **재사용성**이 높아요!

### 상세 페이지의 컴포넌트 분리가 더 정교해졌어요!

```
detail/components/
```

## 3. 개선하면 좋을 점

### 3-1. 상품 목록 페이지가 빌드 설정에 빠져 있어요 (중요도: 높음)

**현재 상태:**

```js
// vite.config.js
build: {
  rollupOptions: {
    input: {
      main: resolve(__dirname, "index.html"),
      productDetail: resolve(__dirname, "src/pages/product/detail/index.html"),
      login: resolve(__dirname, "src/pages/login/index.html"),
      signup: resolve(__dirname, "src/pages/signup/index.html"),
      // ← productList가 없어요!
    },
  },
},
```

**왜 개선하면 좋을까?**

새로 만든 **상품 목록 페이지**(`src/pages/product/list/index.html`)가 빌드 설정에 포함되어 있지 않아요. `npm run dev`(개발 서버)에서는 직접 URL을 입력하면 접근할 수 있지만, `npm run build`(배포용 빌드)를 실행하면 **이 페이지가 빌드되지 않아서 배포 후 접근이 안 돼요!**

이전 리뷰에서 `vite.config.js` 빌드 경로를 잘 수정했는데, 새 페이지를 추가할 때 여기도 함께 추가하는 것을 잊은 거예요. 새 페이지를 만들 때마다 `vite.config.js`에도 등록하는 습관을 들이면 좋아요!

**추천:**

```js
// 수정 전
input: {
  main: resolve(__dirname, "index.html"),
  productDetail: resolve(__dirname, "src/pages/product/detail/index.html"),
  login: resolve(__dirname, "src/pages/login/index.html"),
  signup: resolve(__dirname, "src/pages/signup/index.html"),
},

// 수정 후 — productList 추가
input: {
  main: resolve(__dirname, "index.html"),
  productDetail: resolve(__dirname, "src/pages/product/detail/index.html"),
  productList: resolve(__dirname, "src/pages/product/list/index.html"),
  login: resolve(__dirname, "src/pages/login/index.html"),
  signup: resolve(__dirname, "src/pages/signup/index.html"),
},
```

### 3-2. HTML에서 같은 `id`가 여러 곳에서 사용돼요 (중요도: 높음)

**현재 상태:**

상품 상세 페이지에서 5개의 HTML 파일을 `fetch()`로 불러와서 하나의 페이지에 합치고 있는데, 여러 파일에서 **같은 `id`**를 사용하고 있어요:

```html
<!-- detail-main.html 75번째 줄 -->
<div id="rating" class="flex items-center gap-2">
<div id="rating-stars" class="flex">
<span id="review-count" class="underline text-sm">113</span>

<!-- detail-best-review.html 5번째 줄 -->
<div id="rating-stars" class="flex">         <!-- ← 중복! -->

<!-- detail-review.html 20번째 줄 -->
<div id="rating" class="flex flex-row gap-16 ...">   <!-- ← 중복! -->
<div id="rating-stars" class="flex">                  <!-- ← 중복! -->
<span id="review-count" class="leading-6 ...">113</span>  <!-- ← 중복! -->
```

**왜 개선하면 좋을까?**

HTML에서 `id`는 페이지 전체에서 **딱 하나만** 존재해야 해요. 이것을 "유일성(Unique)"이라고 해요.

같은 `id`가 여러 개 있으면:
- `document.querySelector("#rating-stars")`가 **항상 첫 번째 것만** 찾아요 — 나머지는 무시돼요!
- 나중에 JavaScript로 리뷰 별점을 동적으로 바꾸려고 하면, 의도하지 않은 곳이 바뀔 수 있어요
- HTML 유효성 검사(W3C Validator)에서 오류로 잡혀요

**추천:**

각 영역별로 **고유한 접두어(prefix)**를 붙여주세요:

```html
<!-- detail-main.html — "product-" 접두어 -->
<div id="product-rating" class="flex items-center gap-2">
<div id="product-rating-stars" class="flex">
<span id="product-review-count" class="underline text-sm">113</span>

<!-- detail-best-review.html — "best-review-" 접두어 -->
<div id="best-review-stars" class="flex">

<!-- detail-review.html — "all-review-" 접두어 -->
<div id="all-review-rating" class="flex flex-row gap-16 ...">
<div id="all-review-stars" class="flex">
<span id="all-review-count" class="leading-6 ...">113</span>
```

> **쉽게 말하면:** 같은 반에 "민수"가 3명이면 선생님이 "민수야!"라고 부를 때 누구를 부르는 건지 모르는 것과 같아요. "1반 민수", "2반 민수", "3반 민수"처럼 구분해주면 돼요.

### 3-3. 세로형 상품 카드에서 할인율이 0일 때 "0%"가 표시돼요 (중요도: 높음)

**현재 상태:**

```js
// src/pages/product/list/index.js — 상품 데이터
{
  originalPrice: 29000,
  discountRate: 0,          // ← 할인 없음
  discountPrice: 29000,     // ← 원가와 동일
}
```

```js
// src/components/ui/product-card.js 186~187번째 줄 (세로형)
const discountRateEl = document.createElement("span");
discountRateEl.textContent = `${discountRate}%`;    // ← "0%" 표시!
discountRateEl.className = "text-sm text-burnt-orange font-semibold";

const discountPriceEl = document.createElement("span");
discountPriceEl.textContent = `${discountPrice.toLocaleString()}원`;
// ← discountPrice가 null이면 에러!
```

반면, **가로형**에서는 `if (discountRate)`로 잘 체크하고 있어요:

```js
// product-card.js 75번째 줄 (가로형)
if (discountRate) {     // ← 0이면 건너뛰기 — 잘 했어요!
  const discountRateEl = document.createElement("span");
  // ...
}
```

**왜 개선하면 좋을까?**

JavaScript에서 `0`은 **falsy**(거짓으로 취급되는 값)예요. 그래서 `if (discountRate)`는 할인율이 `0`이면 `false`로 처리돼요. 가로형에서는 이 특성을 잘 활용해서 `0%`가 표시되지 않지만, 세로형에서는 이 체크가 빠져 있어서 **"0%"가 화면에 표시**돼요.

또한 `discountPrice`가 `null`이면 `null.toLocaleString()`에서 **JavaScript 에러**가 나서 페이지가 깨질 수 있어요.

**추천:**

세로형에서도 가로형처럼 할인율 체크를 추가하세요:

```js
// 수정 전 (세로형, 185~193번째 줄)
const discountRateEl = document.createElement("span");
discountRateEl.textContent = `${discountRate}%`;
discountRateEl.className = "text-sm text-burnt-orange font-semibold";

const discountPriceEl = document.createElement("span");
discountPriceEl.textContent = `${discountPrice.toLocaleString()}원`;
discountPriceEl.className = "text-sm text-woody-brown";

discountWrapper.append(discountRateEl, discountPriceEl);

// 수정 후 — 할인 여부에 따라 다르게 표시
if (discountRate) {
  const discountRateEl = document.createElement("span");
  discountRateEl.textContent = `${discountRate}%`;
  discountRateEl.className = "text-sm text-burnt-orange font-semibold";
  discountWrapper.append(discountRateEl);
}

const discountPriceEl = document.createElement("span");
discountPriceEl.textContent = `${(discountPrice ?? originalPrice).toLocaleString()}원`;
discountPriceEl.className = "text-sm text-woody-brown";
discountWrapper.append(discountPriceEl);
```

### 3-4. `renderProductCards` 함수가 두 곳에 중복돼요 — 결합도를 낮추는 방법 (중요도: 중간)

**현재 상태:**

```js
// src/components/ui/product-card-list.js
import { createProductCard } from "/src/components/ui/product-card.js";

export function renderProductCards(products) {
  const container = document.querySelector("#product-list-container");
  container.innerHTML = "";
  products.forEach((product) => {
    container.appendChild(createProductCard(product));
  });
}
```

```js
// src/pages/product/list/index.js
import { createProductCard } from "/src/components/ui/product-card.js";

function renderProductCards(products) {       // ← 거의 똑같은 함수!
  if (!container) return;
  container.innerHTML = "";
  products.forEach((product) => {
    container.appendChild(createProductCard(product));
  });
}
```

**왜 개선하면 좋을까?**

같은 기능을 하는 코드가 두 군데에 있으면:
- 버그를 고칠 때 **두 곳 다** 고쳐야 해요 (하나를 놓치면 한쪽만 고쳐져요)
- 어느 쪽이 "진짜"인지 나중에 헷갈려요
- `product-card-list.js`를 따로 만들어두고 정작 사용하지 않고 있어요!

이것이 바로 **높은 결합도(High Coupling)**의 문제예요. `product-card-list.js`를 만들어놓고 `list/index.js`에서 import하지 않으면, 두 파일이 **독립적으로 같은 일을 하는 셈**이에요.

**추천:**

`list/index.js`에서 이미 만들어둔 `product-card-list.js`를 import해서 사용하세요:

```js
// 수정 전 (src/pages/product/list/index.js)
import { createProductCard } from "/src/components/ui/product-card.js";

function renderProductCards(products) {
  // ... 중복 코드
}

// 수정 후 — 이미 만들어둔 컴포넌트를 재사용!
import { renderProductCards } from "/src/components/ui/product-card-list.js";

// renderProductCards 함수를 직접 만들 필요 없이 바로 사용
renderProductCards(productDataList);
```

> **쉽게 말하면:** 같은 요리를 두 번 만드는 대신, 한 번 만들어서 나눠 먹는 거예요!

### 3-5. 상품 목록 페이지에 `<main>` 태그가 없어요 (중요도: 중간)

**현재 상태:**

```html
<!-- src/pages/product/list/index.html -->
<body>
  <section class="relative w-full h-119 ...">          <!-- 히어로 배너 -->
  </section>
  <div class="px-[232.5px] pt-10 flex flex-col">       <!-- 콘텐츠 -->
    <!-- ... -->
  </div>
</body>
```

다른 페이지들은 `<main>` 태그를 잘 사용하고 있어요:

```html
<!-- index.html -->
<main class="container mx-auto p-4"></main>

<!-- login/index.html -->
<main>...</main>

<!-- signup/index.html -->
<main class="flex justify-center items-center">...</main>
```

**왜 개선하면 좋을까?**

`<main>` 태그는 "이 페이지의 **주요 콘텐츠가 여기에 있어요**"라고 브라우저와 스크린리더에게 알려주는 **시맨틱 태그**예요. `<main>`이 없으면:
- 스크린리더 사용자가 "본문으로 바로가기" 기능을 사용할 수 없어요
- 검색 엔진이 페이지의 핵심 콘텐츠를 구분하기 어려워요
- 다른 페이지와 **일관성**이 깨져요

**추천:**

```html
<!-- 수정 전 -->
<body>
  <section class="relative w-full h-119 ...">
  </section>
  <div class="px-[232.5px] pt-10 flex flex-col">

<!-- 수정 후 — <main>으로 감싸기 -->
```

### 3-6. 상품 목록 페이지의 패딩이 고정값이라 작은 화면에서 깨져요 (중요도: 중간)

**현재 상태:**

```html
<!-- src/pages/product/list/index.html 20번째 줄 -->
<div class="px-[232.5px] pt-10 flex flex-col">
```

**왜 개선하면 좋을까?**

`px-[232.5px]`는 **좌우 양쪽에 232.5px씩** 패딩을 넣는다는 뜻이에요. 화면 너비가 최소 465px 이상이어야 콘텐츠가 보이기 시작하고, 실제로 1000px 미만 화면에서는 콘텐츠 영역이 너무 좁아져요.

다른 페이지들은 반응형을 잘 적용하고 있어요. 예를 들어 로그인 페이지에서 `md:my-18`, `md:px-9.25`를 사용한 것처럼요!

**추천:**

```html
<!-- 수정 전 -->
<div class="px-[232.5px] pt-10 flex flex-col">

<!-- 수정 후 — 반응형 패딩 + max-width 조합 -->
<div class="px-6 md:px-16 lg:px-32 pt-10 flex flex-col max-w-7xl mx-auto">
```

이렇게 하면:
- **모바일** (`sm:`): 좌우 24px 패딩 → 콘텐츠가 충분히 보여요
- **태블릿** (`md:`): 좌우 64px 패딩
- **데스크톱** (`lg:`): 좌우 128px 패딩
- `max-w-7xl mx-auto`: 아주 넓은 화면에서도 중앙 정렬

### 3-7. 카테고리 링크들의 Tailwind 클래스가 6번 반복돼요 (중요도: 중간)

**현재 상태:**

```html
<!-- src/pages/product/list/index.html 37~79번째 줄 -->
<ul class="flex gap-1.5 py-1 px-13">
  <li>
    <a href="" class="bg-spring-wood text-[18px] font-normal leading-5 border-[1px] border-empress rounded-[4px] text-woody-brown text-[12px] whitespace-nowrap py-[6px] px-[10px]">핸드 케어 전체 보기</a>
  </li>
  <li>
    <a href="" class="bg-spring-wood text-[18px] font-normal leading-5 border-[1px] border-empress rounded-[4px] text-woody-brown text-[12px] whitespace-nowrap py-[6px] px-[10px]">핸드 크림</a>
  </li>
  <!-- ... 같은 클래스가 6번 반복! -->
</ul>
```

또한 `text-[18px]`과 `text-[12px]`이 **동시에** 적용되어 있어요. 둘 다 `font-size`를 설정하는데 나중에 선언된 쪽이 이기기 때문에 `text-[18px]`은 무시돼요.

그리고 `href=""`(빈 링크)는 클릭하면 **현재 페이지가 새로고침**돼요.

**왜 개선하면 좋을까?**

1. **같은 스타일을 6번 반복**하면, 스타일을 바꿀 때 6곳을 다 고쳐야 해요 (하나라도 놓치면 UI가 깨져요)
2. `text-[18px]`과 `text-[12px]` **충돌**로 의도하지 않은 글자 크기가 적용될 수 있어요
3. `href=""`는 사용자가 클릭했을 때 혼란스러워요

**추천:**

JavaScript로 카테고리 목록을 동적으로 생성하면 코드 중복을 줄일 수 있어요:

```html
<!-- 수정 전: HTML에 6번 반복 -->
<ul class="flex gap-1.5 py-1 px-13">
  <li><a href="" class="매우 긴 클래스...">핸드 케어 전체 보기</a></li>
  <li><a href="" class="매우 긴 클래스...">핸드 크림</a></li>
  <!-- ... -->
</ul>

<!-- 수정 후: CSS 클래스로 공통 스타일을 묶고, 충돌 제거 -->
<ul id="category-list" class="flex gap-1.5 py-1 px-13">
  <li>
    <a href="#" class="bg-spring-wood text-xs font-normal leading-5 border border-empress rounded py-1.5 px-2.5 text-woody-brown whitespace-nowrap">핸드 케어 전체 보기</a>
  </li>
  <!-- ... -->
</ul>
```

변경 포인트:
- `text-[18px]` 제거 (충돌하는 `text-[12px]`만 남김 → `text-xs`로 통일)
- `border-[1px]` → `border` (Tailwind 기본이 1px이에요)
- `rounded-[4px]` → `rounded` (Tailwind 기본이 4px이에요)
- `py-[6px] px-[10px]` → `py-1.5 px-2.5` (Tailwind 유틸리티 사용)
- `href=""` → `href="#"` 또는 실제 링크 (새로고침 방지)

### 3-9. `--color-empress: #7E717`이 여전히 5자리예요 (중요도: 높음 — 1차 리뷰부터 미해결)

```css
--color-empress: #7E717;    /* ← 5자리! 6자리여야 해요 */
```

**추천:**

```css
/* 수정 전 */
--color-empress: #7E717;

/* 수정 후 — 6자리로 수정 (디자인 시안 확인 필요) */
```

이전 리뷰 14개 항목 중 **10개를 해결**한 것은 정말 대단해요! 특히:

## 2. 잘한 점

### 협업 체계가 훌륭해요!

README.md에 브랜치 전략, 커밋 메시지 규칙, 이슈 라벨, PR 규칙까지 꼼꼼하게 정리했어요. 팀 프로젝트에서 이런 규칙을 처음부터 문서화해두는 건 정말 중요해요.

```
main
  └── develop
        └── feature/#이슈번호-작업내용
```

Git Flow 브랜치 전략을 잘 이해하고 적용하고 있어요. 특히 "본인 PR 본인 머지 금지 — 팀원 1명 이상 리뷰 후 머지" 규칙은 실무에서도 사용하는 좋은 규칙이에요!

### Issue 템플릿 추가가 세심해요

```yaml
# bug_report.md
name: 버그 리포트
about: 버그 발견 시 사용
title: "bug: "
```

버그 리포트 템플릿에 재현 방법, 예상 동작, 실제 동작, 환경 정보를 포함시킨 것은 실무 수준의 구성이에요. 기능 구현 템플릿에 완료 조건 체크리스트와 관련 Epic 연결도 잘 되어 있어요.

### Tailwind v4 @theme 커스텀 색상 시스템

```css
@theme {
  --color-rose-white: #fffefe;
  --color-spring-wood: #fbf9f6;
  --color-merino: #f7f2ea;
  /* ... */
}
```

Tailwind v4의 `@theme`를 활용해서 프로젝트 전용 색상을 정의한 것이 좋아요! 이렇게 하면 `bg-rose-white`, `text-woody-brown`처럼 Tailwind 유틸리티 클래스로 바로 사용할 수 있어요. 색상 이름도 의미가 잘 드러나게 지었어요.

### @layer base로 기본 스타일 설정

```css
@layer base {
  body {
    @apply bg-[var(--color-spring-wood)];
    @apply text-[var(--color-woody-brown)];
  }
}
```

모든 페이지에 공통으로 적용될 기본 배경색과 글자색을 `@layer base`로 설정한 것이 좋아요. 이렇게 하면 매번 `<body>`에 클래스를 넣지 않아도 돼요.

### 로그인 페이지 — 접근성 배려가 잘 되어 있어요

```html
<label for="email">아이디* </label>
<input type="text" id="email" name="user_email" required />
```

`<label>`의 `for`와 `<input>`의 `id`를 올바르게 연결했어요. 이렇게 하면 라벨을 클릭해도 입력창에 포커스가 가요. `required` 속성도 잘 넣어서 빈 값 제출을 막고 있어요. `alt` 속성도 이미지에 잘 적용했어요.

### 로그인 비밀번호 토글 기능 구현

```html
<button type="button" id="togglePassword">
  <img src="./../assets/icon/passwordEye.svg" alt="비밀번호 보기" />
</button>
```

비밀번호 보기/숨기기 토글 버튼을 `type="button"`으로 올바르게 설정했어요. `type="button"`이 없으면 form 안의 버튼이 submit으로 동작해서 페이지가 새로고침될 수 있거든요. 이 부분을 신경 쓴 것이 좋아요!

### 이전 리뷰 피드백 반영 중!

`refactor/index-markup` 브랜치에서 이전 리뷰의 개선 항목을 적극적으로 반영하고 있어요:
- CSS 이중 로딩 → `<link>` 태그 제거 완료
- 시맨틱 태그 → `<header>`, `<footer>` 적용 완료
- 들여쓰기 정리

리뷰를 받고 실제로 개선하는 과정이 가장 중요한 학습이에요. 잘하고 있어요!

### API 문서 정리

`docs/api/` 디렉토리에 인증, 상품, 장바구니, 주문, 리뷰 등 11개의 API 문서를 꼼꼼하게 정리해두었어요. API를 먼저 이해하고 문서화한 후에 구현에 들어가는 것은 매우 좋은 접근이에요!

## 3. 개선하면 좋을 점

### 3-1. `style.css` — 잘못된 색상 코드가 있어요 (중요도: 높음)

**현재 상태:**

```css
/* src/css/style.css 16번째 줄 */
--color-empress: #7E717;
```

**왜 개선하면 좋을까?**

CSS에서 HEX 색상 코드는 반드시 **3자리(`#FFF`) 또는 6자리(`#FFFFFF`)**여야 해요. `#7E717`은 **5자리**라서 유효하지 않은 색상이에요. 브라우저가 이 색상을 무시하거나 예상치 못한 결과를 보여줄 수 있어요.

**추천:**

의도한 색상이 무엇인지 확인하고 6자리로 수정하세요.

```css
/* 수정 전 */
--color-empress: #7E717;

/* 수정 후 — 아래 중 의도한 색상을 선택하세요 */
--color-empress: #7E7177;  /* 앞에 7을 추가한 경우 */
--color-empress: #7E7170;  /* 뒤에 0을 추가한 경우 */
```

### 3-2. `style.css` — CSS 변수 이름에 하이픈이 두 개예요 (중요도: 높음)

**현재 상태:**

`feature/#5-login` 브랜치의 `src/css/style.css`에 이런 변수가 있어요:

```css
/* 로그인 배경1(전체배경) */
--color--grey-99: #fffefa;   /* ← 하이픈이 두 개! (color 다음에 --) */
```

같은 파일 아래쪽에 또 이런 변수가 있어요:

```css
/* 로그인 input */
--color-grey-99: #fffefa;    /* ← 하이픈이 한 개 (정상) */
```

**왜 개선하면 좋을까?**

`--color--grey-99`(하이픈 두 개)와 `--color-grey-99`(하이픈 한 개)는 **서로 다른 변수**예요! Tailwind v4에서 `@theme` 안의 `--color-*` 변수는 자동으로 유틸리티 클래스가 만들어지는데:
- `--color-grey-99` → `bg-grey-99`, `text-grey-99` (정상)
- `--color--grey-99` → `bg--grey-99` (이상한 이름, 사용 어려움)

게다가 값도 같은 `#fffefa`이므로 중복이에요.

**추천:** `--color--grey-99`(하이픈 두 개)를 삭제하세요.

```css
@theme {
  /* 삭제: --color--grey-99: #fffefa; ← 이 줄 제거 */

  /* 이 변수 하나만 유지 */
  --color-grey-99: #fffefa;
}
```

### 3-3. `style.css` — 같은 색상값이 다른 이름으로 중복돼요 (중요도: 중간)

**현재 상태:**

develop 브랜치의 `style.css`:
```css
--color-woody-brown: #3f2b2e;
```

`feature/#5-login` 브랜치에서 추가한 변수:
```css
--color-brand-woody: #3f2b2e;
```

**왜 개선하면 좋을까?**

`--color-woody-brown`과 `--color-brand-woody`가 정확히 같은 색상(`#3f2b2e`)이에요. 같은 색상이 두 개의 이름으로 존재하면 나중에 색상을 바꿀 때 하나만 바꾸고 하나를 깜빡할 수 있어요. 팀원끼리 "이 색상은 woody-brown이야? brand-woody야?" 혼란이 생길 수도 있어요.

**추천:** 하나의 이름으로 통일하세요. 팀에서 합의해서 하나를 정하면 돼요.

```css
@theme {
  /* 둘 중 하나만 남기기 — 팀에서 합의! */
  --color-woody-brown: #3f2b2e;
  /* --color-brand-woody: #3f2b2e;  ← 삭제 */
}
```

### 3-4. 로그인 페이지 — 모바일에서 화면이 깨질 수 있어요 (중요도: 중간)

**현재 상태:**

```html
<!-- feature/#5-login: src/pages/login.html -->
<div class="relative pr-2.5 bg-grey-99 z-10 w-200 h-fit">
  <!-- ... -->
  <div class="my-[71.99px] w-160 mx-9.25 flex flex-col ...">
```

**왜 개선하면 좋을까?**

`w-200`은 Tailwind에서 **800px 고정 너비**예요. 대부분의 스마트폰 화면은 360~430px 정도이므로, 모바일에서 이 모달이 화면 밖으로 삐져나가요. `w-160`(640px)도 마찬가지예요.

**추천:** 고정 너비 대신 최대 너비(`max-w-*`)와 비율 너비(`w-full`)를 조합하세요.

```html
<!-- 수정 전 -->
<div class="relative pr-2.5 bg-grey-99 z-10 w-200 h-fit">

<!-- 수정 후 — 모바일에서도 잘 보여요 -->
<div class="relative pr-2.5 bg-grey-99 z-10 w-full max-w-[800px] h-fit">
```

```html
<!-- 수정 전 -->
<div class="my-[71.99px] w-160 mx-9.25 flex flex-col ...">

<!-- 수정 후 -->
<div class="my-10 md:my-[71.99px] w-full max-w-[640px] mx-auto px-4 flex flex-col ...">
```

### 3-5. 로그인 페이지 — `<form action="">`이 비어 있어요 (중요도: 중간)

**현재 상태:**

```html
<!-- feature/#5-login: src/pages/login.html 41번째 줄 -->
<form action="">
```

**왜 개선하면 좋을까?**

`action=""`은 폼을 제출하면 **현재 페이지 URL로 GET 요청**을 보내요. 로그인 폼은 API 서버로 POST 요청을 보내야 하므로, JavaScript로 처리하거나 action을 제거해야 해요.

**추천:** JavaScript로 폼 제출을 처리할 거라면 `action`을 제거하고 `submit` 이벤트를 막아주세요.

```html
<!-- 수정 전 -->
<form action="">

<!-- 수정 후 — JS로 처리할 때 -->
<form id="loginForm">
```

```js
// src/js/login.js — 새로 만들 파일
const loginForm = document.getElementById("loginForm");
loginForm.addEventListener("submit", (e) => {
  e.preventDefault(); // 페이지 새로고침 방지
  // 여기서 API 호출
});
```

### 3-6. 로그인 페이지 — `<body>` 기본 스타일이 없어요 (중요도: 낮음)

**현재 상태:**

```html
<!-- feature/#5-login: src/pages/login.html -->
<body>
  <main>
```

index.html에서는:
```html
<body class="bg-gray-50 text-gray-900">
```

**왜 개선하면 좋을까?**

`@layer base`에서 `body`에 기본 스타일을 설정했지만, login.html에서 `main.js`를 불러오는 `<script>` 태그는 있어도 CSS가 실제로 적용되는지 확인해야 해요. 또한 index.html과 login.html의 `<body>` 스타일이 다르면 페이지 이동 시 배경색이 갑자기 바뀔 수 있어요.

**추천:** 모든 페이지의 `<body>` 스타일을 통일하거나, `@layer base`의 기본 스타일에 전적으로 맡기세요 (index.html의 `class="bg-gray-50 text-gray-900"`도 제거).

### 3-7. `12-feat-header-footer` 브랜치 — 불필요한 파일이 커밋되었어요 (중요도: 중간)

**현재 상태:**

이 브랜치에 아래 파일들이 커밋되어 있어요:

```
loccishop                          ← git log 출력이 파일로 저장됨
resources/loccitane_sitemap.xml    ← 2,703줄 XML 파일
resources/aesop_sitemap.xml        ← 1,545줄 XML 파일
resources/*.xml:Zone.Identifier    ← Windows Zone Identifier 파일
```

**왜 개선하면 좋을까?**

- `loccishop` 파일은 터미널에서 `git log > loccishop` 같은 명령을 실행했을 때 실수로 생긴 파일이에요.
- sitemap XML 파일은 **참고용 자료**이지 프로젝트 코드가 아니에요. 합쳐서 4,000줄이 넘는 큰 파일이 Git 저장소에 들어가면 다른 팀원들이 `git clone`할 때 불필요하게 오래 걸려요.
- `Zone.Identifier` 파일은 Windows에서 인터넷에서 다운로드한 파일에 자동으로 붙는 메타데이터예요. Git에 올리면 안 되는 파일이에요.

**추천:**

1. `loccishop` 파일과 `Zone.Identifier` 파일을 삭제하세요.
2. sitemap은 참고용이라면 `.gitignore`에 추가하거나 구글 드라이브 등에 따로 공유하세요.
3. `.gitignore`에 추가:

```gitignore
# Windows Zone Identifier
*:Zone.Identifier

# 참고 리소스
resources/
```

### 3-8. `12-feat-header-footer` 브랜치 — header.js, footer.js가 빈 파일이에요 (중요도: 낮음)

**현재 상태:**

```
src/components/header.js    ← 0 bytes (빈 파일)
src/components/footer.js    ← 0 bytes (빈 파일)
```

**왜 개선하면 좋을까?**

파일을 만들어둔 건 좋지만, **빈 파일은 커밋하지 않는 것**이 좋아요. 어떤 내용이 들어갈지 몰라서 다른 팀원이 헷갈릴 수 있어요. 기본 구조라도 넣고 커밋하는 게 좋아요.

**추천:**

```js
// src/components/header.js
export function createHeader() {
  return `
    <nav class="bg-white shadow-md p-4">
      <a href="/" class="text-xl font-bold">L'OcciShop</a>
    </nav>
  `;
}
```

---

### 3-9. `12-feat-header-footer` 브랜치 — 아포스트로피(') 오타 (중요도: 낮음)

**현재 상태:**

```html
<!-- 12-feat-header-footer: index.html -->
<h1>L`OcciShop 클론 프로젝트</h1>
```

**왜 개선하면 좋을까?**

`L'OcciShop`의 아포스트로피(')가 백틱(\`)으로 바뀌어 있어요. 백틱은 프로그래밍에서 쓰는 기호이고, 아포스트로피는 `'` (U+2019) 또는 `'` (U+0027)이에요.

**추천:**

```html
<!-- 수정 전 -->
<h1>L`OcciShop 클론 프로젝트</h1>

<!-- 수정 후 -->
<h1>L'OcciShop 클론 프로젝트</h1>
```

### 3-10. 로그인 브랜치가 두 개 있어요 (중요도: 중간)

**현재 상태:**

같은 로그인 기능을 위한 브랜치가 두 개 존재해요:
- `feature/#5-login`
- `feature/5-loginpage`

두 브랜치의 `login.html`은 동일하고, `style.css`만 약간 다릅니다.

**왜 개선하면 좋을까?**

같은 기능에 대해 브랜치가 두 개 있으면:
- 어떤 브랜치가 최신인지 헷갈려요
- 나중에 둘 다 develop에 머지하면 충돌이 생겨요
- 다른 팀원이 어디에 PR을 보내야 할지 몰라요

**추천:** 팀에서 하나의 브랜치를 정하고 다른 하나는 삭제하세요. README의 브랜치 명명 규칙(`feature/#이슈번호-작업내용`)을 따르면 `feature/#5-login`이 맞아요.
